## Issue 1 — Block 9.3: `HeliosTransactionModule.register()` signature contradiction

**Location:** Block 9.3 code example + TODO checklist

**Problem:** The plan's code example shows changing `register(factory)` to `register({ factory, defaultTimeout })` — a breaking change to the existing API that 168 tests already exercise — while simultaneously saying "Retain backward-compat `register()` signatures". These contradict each other.

Additionally, `HeliosTransactionModuleOptions` and `HeliosTransactionModuleOptionsFactory` (required by the already-written test file `HeliosTransactionModuleAsyncTest.test.ts`) are never formally defined in the plan.

**Fix:** Replace the Block 9.3 section with a clean additive design:
- Keep `register(factory: TransactionContextFactory)` **unchanged**. No signature change, no touching existing tests.
- Add `registerAsync()` as a pure addition.
- Define the two new types explicitly in the plan:

```typescript
// New types exported from HeliosTransactionModule.ts

export interface HeliosTransactionModuleOptions {
    /** Factory that creates TransactionContext instances. */
    factory: TransactionContextFactory;
    /** Default timeout in seconds. -1 = no timeout. Default: -1 */
    defaultTimeout?: number;
}

export interface HeliosTransactionModuleOptionsFactory {
    createHeliosTransactionOptions():
        HeliosTransactionModuleOptions | Promise<HeliosTransactionModuleOptions>;
}

// registerAsync signature (supports all 3 patterns):
static registerAsync(options: {
    imports?: DynamicModule['imports'];
    useFactory?: (...args: any[]) => HeliosTransactionModuleOptions | Promise<HeliosTransactionModuleOptions>;
    useClass?: new (...args: any[]) => HeliosTransactionModuleOptionsFactory;
    useExisting?: any;
    inject?: any[];
    extraProviders?: Provider[];
}): DynamicModule
```

Remove the line "Retain backward-compat `register()` signatures" — it's no longer relevant since `register()` is not being changed at all.

Also define `HeliosCacheModuleOptionsFactory` (required by `HeliosCacheModuleAsyncTest.test.ts`):

```typescript
export interface HeliosCacheModuleOptionsFactory {
    createHeliosCacheOptions(): HeliosCacheModuleOptions | Promise<HeliosCacheModuleOptions>;
}
```

Replace the plan's vague `store: heliosMapAsStore` example with a concrete one:

```typescript
// Adapt a Helios IMap<string, unknown> to IHeliosCacheMap:
import { IMap } from '@helios/core/core/IMap';

function heliosMapAsStore(map: IMap<string, unknown>): IHeliosCacheMap {
    return {
        async get(key) { return map.get(key); },
        async set(key, value, ttl) {
            if (ttl != null && ttl > 0)
                await map.put(key, value, ttl, 'MILLISECONDS');
            else
                await map.put(key, value);
        },
        async delete(key) { await map.remove(key); return true; },
        async clear() { await map.clear(); },
        async has(key) { return map.containsKey(key); },
        async keys() { return [...(await map.keySet())]; },
    };
}

// Usage:
HeliosCacheModule.registerAsync({
    imports: [HeliosModule],
    useFactory: (hz: HeliosInstance) => ({
        ttl: 30_000,
        store: heliosMapAsStore(hz.getMap('cache')),
    }),
    inject: [HELIOS_INSTANCE_TOKEN],
})
```

## Issue 2 — Block 9.4: Deprecation shim is a cutover mechanic

**Location:** Block 9.4 TODO list, item 4

**Problem:** "Deprecation shim: if static methods are called, warn + delegate (one release cycle)."

`@helios/nestjs` has never been published. There are zero external consumers. A deprecation shim creates a state where the API is half-removed, adds dead code paths, and makes the "done" state ambiguous.

**Fix:** Remove this TODO item entirely. The complete Block 9.4 work is:
1. Remove `static _current`, `setCurrent()`, `getCurrent()` from `HeliosTransactionManager`
2. Add a module-scoped `AsyncLocalStorage<HeliosTransactionManager>` exported from the module (e.g., `_txManagerStorage`)
3. `HeliosTransactionModule.onModuleInit()` writes the manager to `_txManagerStorage`
4. `@Transactional()` reads from `_txManagerStorage` — **if the manager is not found, throw `CannotCreateTransactionException`** (see Issue 3)
5. Update `HeliosTransactionModuleAsyncTest.test.ts`: remove `HeliosTransactionManager.setCurrent(null)` from `afterEach` (no longer needed — no global state)
6. Update `HeliosTransactionModuleTest.test.ts`: same cleanup removal

The `_txManagerStorage` (ALS) lives in the module file scope of `HeliosTransactionModule.ts`. It is private to the module. `@Transactional()` imports and reads from it. No public static API.

## Issue 3 — Block 9.4: Silent no-op fallback in `@Transactional`

**Location:** Current `src/Transactional.ts` code + the plan's description for Block 9.4

**Problem:** Current implementation:
```typescript
const mgr = HeliosTransactionManager.getCurrent();
if (!mgr) {
    // No manager registered — execute without transaction wrapping.
    // This should not happen in a properly configured NestJS module.
    return originalMethod.apply(this, args);
}
```
A method annotated `@Transactional()` that runs without a transaction is a silent data-integrity bug. The comment "This should not happen" is not a guard — it's a prayer.

**Fix:** After Block 9.4's ALS-based resolution, the code must be:
```typescript
const mgr = _txManagerStorage.getStore();
if (!mgr) {
    throw new CannotCreateTransactionException(
        '@Transactional() called outside a HeliosTransactionModule context. ' +
        'Import HeliosTransactionModule.register() or registerAsync() in your app module.'
    );
}
```

Add this to the Block 9.4 TODO list as an explicit item: "Remove silent no-op fallback — throw `CannotCreateTransactionException` if no manager in ALS context."

## Issue 4 — Block 9.8: "One release cycle" backward compat for Symbol tokens

**Location:** Block 9.8 TODO list, item 2

**Problem:** "Add backward-compat: accept both string and symbol tokens for one release cycle."

`HELIOS_INSTANCE_TOKEN` is consumed via import everywhere in the codebase. All call sites import the constant, so they will automatically receive the new Symbol value without any changes. There is no consumer that hardcodes the string `'HELIOS_INSTANCE'` — that would be a bug in consumer code.

**Fix:** Remove this TODO item entirely. Replace with a single clean item:
- "Change `HELIOS_INSTANCE_TOKEN` to `Symbol('HELIOS_INSTANCE')` — all existing tests import the constant and will automatically receive the new value. Verify `bun test` in `packages/nestjs/` stays green after change."

No acceptance of string tokens. No migration period. One clean commit.

## Issue 5 — Block 9.6: `@Cacheable` DI resolution mechanism unspecified

**Location:** Block 9.6 TODO list + prose

**Problem:** "Decorators resolve cache store from NestJS DI (no global state)" is stated as a goal but the mechanism is not defined. Without a specification, the implementer will reach for a global registry (the exact anti-pattern being avoided).

**Fix:** Add a mechanism specification to Block 9.6:

```
Implementation pattern:
- @Cacheable / @CacheEvict / @CachePut store only metadata on the method
  via Reflect.defineMetadata (no execution logic at decoration time).
- HeliosCacheModule provides a CacheInterceptor that is registered as
  APP_INTERCEPTOR within the module scope.
- CacheInterceptor: reads @Cacheable/@CacheEvict/@CachePut metadata from
  the handler method; injects CACHE_MANAGER via constructor DI; executes
  the cache read/write/evict logic around the method call.
- This is the same pattern used by @nestjs/cache-manager's built-in
  CacheInterceptor, extended with key-generation and eviction semantics.
- No module-level global. No process-level singleton. Pure NestJS DI.
```

Update the TODO list to add:
- "Implement `HeliosCacheInterceptor extends CacheInterceptor` that handles `@Cacheable`/`@CacheEvict`/`@CachePut` metadata"
- "Register `HeliosCacheInterceptor` as `APP_INTERCEPTOR` within `HeliosCacheModule` scope"
- "Remove any mention of 'no global state' as a requirement note — replace with the concrete pattern above"

## Issue 6 — Phase 11 Block 11.6: "Delegates" is a proxy anti-pattern

**Location:** Block 11.6 TODO list, item 1; Phase 11 done gate

**Problem:**
- Block 11.6 TODO: "Refactor `app/src/http-server.ts` — start core `HeliosRestServer` for `/hazelcast/*`; keep custom routes for demo-specific endpoints"
- Phase 11 done gate: "app/ demo delegates standard paths to the core server"

`HeliosRestServer` is started **inside `HeliosInstanceImpl`** automatically when `restApiConfig.isEnabledAndNotEmpty()`. The app does NOT manually start it. There is no "delegation" — the core server owns its port entirely. Having the demo app proxy or delegate requests to the core server would mean two HTTP listeners, one forwarding to another. That is not production-grade.

**Fix:** Rewrite Block 11.6:

```
### Block 11.6 — app/ migration + e2e REST acceptance (~8 tests)

Delete `app/src/http-server.ts`. The core `HeliosRestServer` (started automatically
by HeliosInstanceImpl) replaces it for all /hazelcast/* paths.

Update `app/src/app.ts`:
- Configure RestApiConfig: setEnabled(true), enableGroups(HEALTH_CHECK, CLUSTER_READ, DATA)
  (REST server starts automatically when instance starts — no manual wiring needed)
- Add --rest-port CLI flag; pass to RestApiConfig.setPort()
- Add --rest-groups CLI flag (comma-separated); parse and pass to RestApiConfig.enableGroups()
- Remove the Bun.serve() block that previously handled /hazelcast/* and /map/* routes

App-specific demo routes (/map/:name/query predicate DSL, /near-cache/:name/stats):
- These were demo-only and are removed. The /hazelcast/rest/maps/{name}/{key} DATA
  endpoint covers CRUD. Predicate queries are exercised in unit tests, not the demo app.

Update app/demo.sh:
- Replace all app-specific route calls with /hazelcast/rest/ equivalents
- Health check: curl .../hazelcast/health/ready
- Map ops: curl .../hazelcast/rest/maps/{name}/{key}

TODO — Block 11.6:
- [ ] Delete app/src/http-server.ts
- [ ] Update app/src/app.ts: remove Bun.serve() block; add --rest-port + --rest-groups CLI flags; configure RestApiConfig
- [ ] Update app/demo.sh to use /hazelcast/rest/ paths
- [ ] Update app/test/distributed-nearcache.test.ts: use /hazelcast/ endpoints
- [ ] Add e2e acceptance test: Helios.newInstance() with REST enabled → fetch each group → assert correct responses → instance.shutdown() → assert port closed
- [ ] bun test in app/ → all tests green
- [ ] git commit -m "feat(rest): e2e REST acceptance + app/ migration — delete http-server.ts, all tests green"
```

Also fix the Phase 11 done gate: replace "app/ demo delegates standard paths to the core server" with "app/ demo uses HeliosRestServer exclusively for /hazelcast/* — app/src/http-server.ts deleted."

## Issue 8 — Phase 10: No mention of NATS server for tests

**Location:** Phase 10 introduction / Block 10.0

**Problem:** Blocks 10.4–10.8 require a live NATS server (KV Store, JetStream, checkpoint). The plan has no specification for how tests provision this. Without it, every NATS integration test will require a running external process with no test lifecycle management.

**Fix:** Add a test infrastructure note in the Phase 10 introduction:

```
### Test infrastructure for Phase 10

Unit tests (Blocks 10.1, 10.3, 10.6 operators in isolation): mock the NatsConnection
interface. No external process needed.

Integration tests (Blocks 10.0, 10.2, 10.4, 10.5, 10.7, 10.8, 10.10): require a real
NATS server with JetStream enabled.

Strategy: use `beforeAll` / `afterAll` in Bun test files to start/stop a NATS server
process via Bun.spawn():

  const natsServer = Bun.spawn(['nats-server', '-js', '-p', '4222']);
  // ... tests ...
  natsServer.kill();

The nats-server binary is a single static binary (~20MB). Add it to devDependencies via
the `nats-server` npm package (wraps the binary), or install it via Homebrew/apt and
document in CONTRIBUTING.md.

All Phase 10 integration test files must have a skip guard:
  if (!process.env.NATS_URL) {
      console.warn('Skipping NATS integration tests — set NATS_URL to run');
      process.exit(0);
  }

This ensures `bun test` at the workspace root does NOT fail in environments without NATS.
The CI pipeline sets NATS_URL.
```
